# Мастер настройки коммутаторов (USB-COM)

Программа для быстрой первичной настройки коммутаторов **без знания CLI**.
Оператор подключает USB-COM переходник, выбирает модель, заполняет поля —
программа сама логинится в консоль, отправляет команды и сохраняет конфиг.

Основная модель — **D-Link DGS-1250-28X/A1A**. Также есть профили для
**Eltex MES**, **TP-Link T-серия**, **Zyxel GS/XGS** и старых **D-Link DES/DGS-3xxx**.

---

## Сборка .exe (делается один раз, на любой машине с Python)

1. Установить Python 3.10+ с python.org, отметив галочку *Add Python to PATH*.
2. Положить все файлы проекта в одну папку.
3. Запустить `build.bat` двойным кликом.
4. Готовый файл: `dist\switch-wizard.exe` + `dist\profiles.yaml`.

Дальше эту папку можно копировать на любой компьютер — Python на нём не нужен.
**Важно:** `profiles.yaml` должен лежать рядом с `.exe`, программа читает его при запуске.

Драйвер USB-COM переходника (Prolific PL2303 / FTDI / CH340) ставится отдельно.

---

## Как пользоваться

1. Подключить переходник к консольному порту коммутатора, выбрать COM-порт
   (кнопка **Обновить**) и модель. Скорость подставится автоматически.
2. **Проверить связь** — программа разбудит консоль и покажет приглашение.
   Если тишина — попробуйте другую скорость: 115200 → 38400 → 9600.
3. Заполнить поля: имя, IP управления, маска, шлюз, VLAN управления, логин и пароль.
4. Списки VLAN и портов — простым текстом:

   ```
   VLAN:                    Порты:
   10 Office                1-20 access 10
   20 VoIP                  21-22 access 20
   30 Guest                 23-24 trunk 10,20,30
   ```

5. **Показать команды** — предпросмотр без отправки. Полезно на этапе обкатки.
6. **НАСТРОИТЬ КОММУТАТОР** — выполнение с прогресс-баром и живым журналом.
7. **Сохранить лог** — файл сессии для отчёта или разбора ошибки.

Встроенные проверки до отправки: корректность IP/маски/шлюза, диапазонов портов,
и что каждый VLAN, назначенный на порт, объявлен в списке VLAN.
При ошибке от коммутатора выполнение останавливается **до** команды сохранения —
достаточно перезагрузить устройство без записи, чтобы вернуть исходное состояние.

## D-Link DGS-1250-28X/A1A — основная модель

Профиль `dlink_dgs1250` стоит первым в списке и выбирается по умолчанию.

Важно: DGS-1250 использует **новый CLI D-Link** («industrial-like», как у
DGS-1510 и DGS-3130) с режимом `configure terminal`, а не старый синтаксис
`create vlan ... tag ...` от DES-3200. Профиль `dlink_legacy` для этой модели
**не подходит** — он оставлен только для старых коммутаторов в парке.

Что учтено под эту модель:

- Консоль **115200**, вход по заводским `admin` / `admin`.
- Порты адресуются как `1/0/1`; используется `interface range ethernet
  1/0/1-1/0/24` — полная настройка 28-портового коммутатора укладывается
  примерно в 32 команды вместо сотни.
- SFP+ порты 25–28 — в поле портов пишутся так же: `25-28 trunk 10,20`.
- Ключ SSH генерируется в привилегированном режиме (`crypto key generate rsa
  modulus 1024`), на это отведено 120 секунд; сохранение конфига — 90 секунд.
- Подтверждения `[y/n]`, `[yes/no]` и `Destination filename startup-config?`
  программа отвечает автоматически.

Два блока закомментированы в профиле намеренно — включите после первого
успешного прогона на тестовом коммутаторе:

- `no username admin` — удаление заводской учётки. Сначала убедитесь, что
  заходите под `root`, иначе рискуете потерять доступ.
- `no snmp-server community public` / `private` — удаление заводских community.
  Если записи отсутствуют, коммутатор вернёт ошибку и мастер остановится.

## Два набора учётных данных

Не путайте поля: **«Вход в коммутатор»** (вверху, рядом с COM-портом) — это
заводские `admin` / `admin`, под которыми программа заходит в консоль.
**«Новый логин» / «Новый пароль»** — учётка `root`, которую программа создаёт.
При перенастройке уже настроенного коммутатора впишите в поля входа `root` и
рабочий пароль.

## Единый корпоративный стандарт

Блок `defaults` в начале `profiles.yaml` задаёт значения, которые подставляются
в поля мастера при каждом запуске:

```yaml
defaults:
  username: "root"
  password: "aaebkgb"
  snmp_ro: "troitsk77"
  snmp_mandatory: true      # галочку «Настроить SNMP» снять нельзя
  lock_credentials: false   # true — запретить оператору править эти три поля
```

Оператору остаётся заполнить только имя, IP, шлюз, VLAN и порты — учётка и
SNMP community применяются на каждом коммутаторе автоматически.
Если поставить `lock_credentials: true`, поля станут доступны только для чтения.

Заводские community `public` / `private` по-хорошему стоит удалять — заготовки
команд лежат закомментированными в секции `snmp` профиля D-Link.

> **О хранении пароля.** Пароль лежит открытым текстом в `profiles.yaml`, то есть
> любой, у кого есть папка с программой, получает доступ ко всем коммутаторам.
> Раздавайте её только исполнителям, держите на сетевом ресурсе с ограниченным
> доступом и меняйте пароль при увольнении сотрудника. Более надёжный вариант на
> будущее — оставить `password: ""` и выдавать пароль оператору отдельно.

## Проверка команд перед боевым использованием (обязательно)

Наборы команд в `profiles.yaml` написаны под типовые прошивки, но у D-Link,
TP-Link и Zyxel синтаксис заметно расходится между сериями и версиями ПО.
Перед запуском в работу прогоните каждую модель на одном тестовом коммутаторе
и поправьте профиль там, где команда не прошла.

Особенно стоит проверить:

- **D-Link DGS-1250** — профиль собран по документации серии, но прошивки
  1.00 и 2.0x различаются в мелочах; обязательно прогоните «Показать команды»
  и первый боевой запуск на тестовом коммутаторе.
- **TP-Link** — режим порта: `general` vs `access`, зависит от серии.
- **Zyxel** — команда включения SSH и формат сохранения (`write memory` / `copy`).
- **Eltex** — на части моделей нужен `vlan database` перед созданием VLAN.

Правки вносятся в `profiles.yaml` обычным текстовым редактором, пересборка
не требуется — достаточно перезапустить программу.

## Как добавить новую модель

Скопируйте любой блок в `profiles.yaml`, поменяйте `title`, `baudrate`,
`iface_template` и команды в секциях. Секции выполняются в порядке:
`enter_config → hostname → vlans → mgmt → user → ssh → snmp → ports_access →
ports_trunk → save`. Пустую секцию можно удалить.

Доступные подстановки: `{{hostname}}`, `{{mgmt_ip}}`, `{{mgmt_mask}}`,
`{{mgmt_prefix}}`, `{{gateway}}`, `{{mgmt_vlan}}`, `{{mgmt_vlan_name}}`,
`{{username}}`, `{{password}}`, `{{snmp_ro}}`, `{{vlan_id}}`, `{{vlan_name}}`,
`{{port}}`, `{{iface}}`, `{{port_range}}`, `{{vlan}}`, `{{vlans_csv}}`.

## Организационные моменты

- Пароль вводится вручную и маскируется в журнале — в файлы он не пишется.
- Есть смысл завести общий сетевой каталог с логами: видно, кто и что настроил.
- Логичное развитие: заранее заготовленные шаблоны объектов («Офис», «Склад»),
  чтобы оператор выбирал один пункт вместо заполнения всех полей, и выгрузка
  готового конфига в бэкап-папку.

## Структура проекта

| Файл | Назначение |
|---|---|
| `profiles.yaml` | Команды по вендорам — **правится админом** |
| `config_builder.py` | Разбор ввода, валидация, сборка списка команд |
| `serial_session.py` | COM-порт, логин, отправка команд, разбор ответов |
| `gui.py` | Интерфейс мастера |
| `main.py` | Точка входа |
| `build.bat` | Сборка .exe |
